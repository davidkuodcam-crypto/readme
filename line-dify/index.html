💡 觀念 1：LINE 的前台與後台（店面 vs. 機房）

很多同學剛接觸 LINE 機器人時，會被兩個不同的管理網址搞得暈頭轉向。我們用「開餐廳」來比喻：

  • LINE 官方帳號管理系統 (Manager) = 餐廳的「接待店面」
    網址：manager.line.biz。
    這裡是老闆日常營運的地方。你可以看報表、發群發廣播、設定基本的「關鍵字回覆」或營業時間。這是一個給一般人用的介面。
  • LINE 開發者控制台 (Developers) = 餐廳的「機房與管線區」
    網址：developers.line.biz。
    這裡是工程師（也就是現在的你）拉電線的地方。你需要來這裡取得「API 金鑰（通關密碼）」，才能把外部的 AI 系統連接到你的店面上。這是一個給開發者用的介面。

💡 觀念 2：LINE、Vercel、Dify 聯手運作的流程

為什麼我們不能直接把 LINE 連到 Dify？因為他們語言不通！為了讓這套系統完美運作，我們必須明確區分出「前端與後台」的角色，把它們想像成一個「四人接力的超級客服團隊」：

LINE 前端 App 客人的嘴巴與耳朵
LINE 後台伺服器 第一線接待櫃台
Vercel 中繼站 Python 翻譯快遞員
Dify 雲端大腦 知識庫與 AI 模型
實戰舉例：當信徒提問「這週日崇拜幾點？」
  1. 【前端 App】客人發問：信徒打開手機裡的 LINE App 輸入問題並傳送。
  2. 【後台伺服器】接待與轉發：LINE 的官方伺服器接收到這個問題。因為我們設定了 Webhook，伺服器會像總機一樣，馬上把這句話打包丟給我們指定的 Vercel 網址。
  3. 【Vercel 中繼站】翻譯：住在 Vercel 裡的 Python 程式收到包裹。因為 LINE 傳來的 JSON 格式 Dify 聽不懂，Python 會把它「翻譯」成 Dify 懂的格式，並附上 Dify 的 API 金鑰，火速送往大腦。
  4. 【Dify 雲端大腦】思考：Dify 收到問題，翻找你之前上傳的教會週報知識庫（RAG），生成了一段精準的回應，交還給快遞員。
  5. 【Vercel 中繼站】重新包裝：Python 程式接過答案，把它重新包裝成 LINE 規定的格式（如果裡面有網址，程式還會幫你變成漂亮的按鈕），然後帶上 LINE 的通行證送回後台。
  6. 【LINE 後台 -> 前端 App】派送回覆：LINE 後台伺服器收到答案，立刻推播回信徒的手機 LINE App。「叮」一聲，信徒看到了完美的 AI 回覆！
1

帳號觀念釐清

在開始打造你的 AI 助理前，我們必須先弄清楚「你」與「機器人」的關係。這也是新手最常混淆的地方。

1. 自然人帳號 vs. 官方帳號

自然人帳號 (登入身分)

就像是一把「鑰匙」。這代表的是你本人（例如：郭大維老師），你是擁有管理員權限的活人。

官方帳號 (專案帳號)

就像是「房間」。這是你建立的機器人或服務（例如：文藻 AI 助教），提供給大眾加入好友。你可以用同一把鑰匙，管理多個不同的房間。

Q：用個人 LINE 帳號建立專案，我的私人 LINE 會被看光嗎？

A：絕對不會。 官方帳號是獨立的服務。你在私人 LINE App 中不會直接看到它，除非你像一般使用者一樣去「掃 QR Code 加入好友」。

2. 登入方式的差異：綠色按鈕 vs. 白色按鈕

比較項目
推薦

🟢 使用 LINE 帳號登入 		⚪ 使用商用帳號登入
綁定對象 你個人的手機 LINE App 你的工作 Email
便利性 極高，掃描 QR Code 即可秒登入 需輸入 Email 與密碼
適用對象 個人開發者、測試階段、中小企業 大型公司、完全不想和私帳沾上邊的人
功能差異 完全相同 完全相同

老師建議： 在開發與測試階段，強烈建議使用「LINE 帳號」登入。這能讓你在測試 API 時，無縫使用手機接收測試結果，免去重複認證的麻煩。

3. 灰盾與藍盾：要申請認證嗎？

  • 灰盾牌（一般帳號） 預設狀態。無法在 LINE 中被直接搜尋到，別人必須有 ID 或 QR Code 才能加。適合開發測試中。
  • 藍盾牌（認證帳號） 需提交組織證明文件審核。可在搜尋欄直接被搜到，具有官方背書的信任感。

4. 如何讓使用者加入？

因為灰盾帳號搜不到，你必須主動提供入口：

  1. 進入左側選單 增加好友人數 > 增加好友工具。
  2. 選擇 建立加入好友行動條碼。
  3. 將此 QR Code 下載或印出，或複製下方網址連結。
2

系統大腦設定

要讓 LINE 帳號從「智障型」變成「智慧型」，關鍵就在這兩個開關。

Messaging API

這是一套讓「外部程式」能操控 LINE 通訊管道的工具。開啟它，你才能寫程式控制這個帳號。

Webhook

這是一個「反向通知」機制：

  • 關閉時： 全靠後台設定的「自動回覆」來應答。
  • 開啟時： 呼叫你寫的程式 (如 Python)，讓 AI 接管對話。
3

內建自動回應訊息解析

路徑：在 設定 > 回應設定 中，選擇 「手動聊天 + 自動回應訊息」 模式。

LINE 的自動回覆分為兩種邏輯，千萬別搞混了：

1. 關鍵字回應 Sniper 狙擊手模式

  • 觸發條件： 使用者傳送的訊息 「完全符合」 你設定的關鍵字。
  • 設定陷阱： 輸入詞彙後 必須按下 Enter 鍵，讓它變成一個「藍色標籤框框」才算成功。若設定為「聚會時間」，使用者傳「請問聚會時間」，則 不會 觸發（因為多了字）。
  • 用途： 針對特定問題給出精確答案（如：地址、匯款帳號、營業時間）。

2. 一律回應 / Default Goalkeeper 守門員模式

  • 觸發條件： 使用者傳送的訊息 「沒有命中」 任何關鍵字時，系統就會掉入這個墊底的回覆。
  • 用途： 當作客服導引。例如：
    「平安！感謝您的訊息。若是關於聚會時間或地點，請輸入關鍵字查詢。」
4

串接 Dify AI Agent 實戰架構

當你準備好將 LINE 帳號交由 Dify 上建立的 AI Agent 來接管時，這是一個重大的架構轉變。

核心疑問解答

Q：改由 Dify 回答後，剛才設定的「關鍵字」和「一律回應」都會失效嗎？

是的，這取決於你如何設定 Webhook 開關。

當你把 Webhook 打開 (啟用) 時，LINE 會將使用者的每一則訊息都 優先轉發 給你的程式 (也就是轉發給 Dify)。 此時，如果你在後台的「自動回應訊息」選項中：

  • 如果你將回應方式改為 單純的「手動聊天」：那麼之前設定的關鍵字和一律回應會 完全失效，帳號將 100% 由 Dify 接管。
  • 如果你 保持勾選 「手動聊天 + 自動回應訊息」：這會造成 災難性的「雙重回覆」！使用者傳送一個關鍵字，他會同時收到 Dify 的 AI 回覆，以及 LINE 內建的關鍵字回覆。

5

實戰演練：部署中繼站程式碼

LINE 無法直接聽懂 Dify 的語言。我們將使用 Python (FastAPI) 寫一個中繼站（翻譯官），並免費部署到 Vercel 伺服器上。

Step 1 取得 LINE 與 Dify 的通關鑰匙 (API Keys)

LINE Developers 必備的兩把鑰匙

LINE_CHANNEL_ACCESS_TOKEN

這是什麼？（存取權杖）
機器人「發言」的通行證。我們的程式要回傳文字或按鈕給使用者時，必須附上這串憑證，LINE 伺服器才會放行。

LINE_CHANNEL_SECRET

這是什麼？（頻道密碼）
用來「驗證身分」的防偽密碼。程式會用它來檢查收到的訊息是不是真的來自 LINE 官方，防止駭客惡意偽造訊息。

✅ 串接已建立的官方帳號並取得鑰匙：
  1. 1. 啟用 API 請先回到 LINE 官方帳號管理後台，進入你已經建好的帳號（例如：新高台福教會）。點擊右上角的 「設定」 > 左側選單的 「Messaging API」，然後點擊 「啟用 Messaging API」。
  2. 2. 綁定開發者 系統會要求選擇「服務提供者 (Provider)」。在下拉選單中，直接選擇你剛才建立的 Provider（如果找不到，就直接在框框裡輸入名稱新建一個），接著按確定完成綁定。
  3. 3. 取得 Secret 啟用完成後，點擊頁面上的連結進入 LINE Developers 後台。停留在 Basic settings (基本設定) 分頁，往下捲動找到 Channel secret，點擊右側的複製按鈕。
  4. 4. 取得 Token 點擊上方的 Messaging API 分頁，往下捲動到底部找到 Channel access token (long-lived)。請點擊 Issue (發行)，然後複製那串超長代碼。

Dify 的大腦鑰匙

DIFY_API_KEY

這是什麼？ 這是呼叫 AI 幫你思考與回答的通關密碼。沒有它，中繼站就無法向 Dify 提問。

✅ 取得步驟：
  • 進入 Dify 後台，點擊進入你的 Agent 應用程式。
  • 點擊左側選單或右上角的 「存取 API (API Access)」。
  • 點擊右上角的「API 金鑰」，建立一組新的金鑰並複製 (開頭通常是 app-)。
小提醒： 請先打開電腦的「記事本」，將這三把鑰匙貼上去保管好，我們在 Step 3 部署伺服器時馬上就會用到！

Step 2 建立專案與 4 個關鍵檔案

資料夾結構非常重要！

因為 Vercel 的部署特性，我們必須建立一個名為 api 的資料夾，並將主程式放進去。你的專案資料夾長相必須如下：

📄 api/main.py 這是我們的大腦中繼站 
from fastapi import FastAPI, Request, Header, HTTPException
import requests
import os
import hmac
import hashlib
import base64
import re
import json

app = FastAPI()

LINE_ACCESS_TOKEN = os.getenv('LINE_CHANNEL_ACCESS_TOKEN')
LINE_SECRET = os.getenv('LINE_CHANNEL_SECRET')
DIFY_API_KEY = os.getenv('DIFY_API_KEY')

# 【新增】用來儲存每個使用者的對話 ID (記憶功能)
# 注意：若部署在 Vercel 免費版，一段時間沒人使用伺服器休眠時，這個記憶會被清空。
# 若需永久記憶，建議改存入資料庫 (如 Redis、Firebase 或 Vercel KV)
user_sessions = {}

@app.post("/webhook")
async def callback(request: Request, x_line_signature: str = Header(None)):
    body = await request.body()
    body_str = body.decode("utf-8")

    # 1. 驗證這條訊息真的是 LINE 官方傳來的 (資安防護)
    if LINE_SECRET and x_line_signature:
        hash_val = hmac.new(LINE_SECRET.encode('utf-8'), body_str.encode('utf-8'), hashlib.sha256).digest()
        signature = base64.b64encode(hash_val).decode('utf-8')
        if signature != x_line_signature:
            raise HTTPException(status_code=400, detail="Invalid signature")

    # 2. 解析訊息並處理
    data = await request.json()
    for event in data.get('events', []):
        if event.get('type') == 'message' and event.get('message', {}).get('type') == 'text':
            user_message = event['message']['text']
            reply_token = event['replyToken']
            user_id = event['source'].get('userId', 'unknown_user')

            # --- 步驟 A：準備呼叫 Dify 大腦的資料 ---
            dify_payload = {
                "inputs": {},
                "query": user_message,
                "response_mode": "streaming", # Agent 必須使用串流模式
                "user": user_id
            }
            
            # 【新增】如果這個使用者之前有對話過，就把 conversation_id 帶上
            if user_id in user_sessions:
                dify_payload["conversation_id"] = user_sessions[user_id]

            try:
                # 傳送請求給 Dify
                dify_res = requests.post(
                    "https://api.dify.ai/v1/chat-messages",
                    headers={"Authorization": f"Bearer {DIFY_API_KEY}"},
                    json=dify_payload,
                    stream=True
                )
                
                # 檢查連線狀態
                if dify_res.status_code != 200:
                    error_data = dify_res.json()
                    error_msg = error_data.get('message', str(error_data))
                    answer = f"⚠️ Dify 大腦回報錯誤：\n{error_msg}"
                else:
                    answer = ""
                    # 逐行讀取串流資料
                    for line in dify_res.iter_lines():
                        if line:
                            decoded_line = line.decode('utf-8')
                            if decoded_line.startswith('data:'):
                                data_str = decoded_line[5:].strip()
                                try:
                                    json_data = json.loads(data_str)
                                    
                                    # 【新增】抓取並儲存 conversation_id，讓下一次對話能接續
                                    if 'conversation_id' in json_data:
                                        user_sessions[user_id] = json_data['conversation_id']
                                        
                                    # 兼容 Agent 模式與 Chatflow 模式
                                    event_type = json_data.get('event')
                                    if event_type in ['message', 'agent_message']:
                                        answer += json_data.get('answer', '')
                                except json.JSONDecodeError:
                                    pass
                                    
                    if not answer:
                        answer = "Dify 處理完畢，但未產生文字回應 (可能只回傳了思考過程)。"
                        
            except Exception as e:
                answer = f"伺服器連線例外錯誤：{str(e)}"

            # --- 步驟 B：超連結轉按鈕邏輯 (Flex Message) ---
            links = re.findall(r'\[([^\]]+)\]\((https?://[^\)]+)\)', answer)
            
            if links:
                clean_text = re.sub(r'\[([^\]]+)\]\((https?://[^\)]+)\)', r'\1 (請點擊下方按鈕)', answer)
                buttons = []
                for title, url in links[:4]:
                    buttons.append({
                        "type": "button",
                        "style": "primary",
                        "margin": "sm",
                        "action": {
                            "type": "uri",
                            "label": title[:20],
                            "uri": url
                        }
                    })
                
                messages = [{
                    "type": "flex",
                    "altText": "助教傳送了一個連結給您",
                    "contents": {
                        "type": "bubble",
                        "body": {
                            "type": "box",
                            "layout": "vertical",
                            "contents": [
                                {
                                    "type": "text",
                                    "text": clean_text,
                                    "wrap": True
                                }
                            ] + buttons
                        }
                    }
                }]
            else:
                messages = [{"type": "text", "text": answer}]

            # --- 步驟 C：傳回 LINE ---
            requests.post(
                "https://api.line.me/v2/bot/message/reply",
                headers={
                    "Content-Type": "application/json",
                    "Authorization": f"Bearer {LINE_ACCESS_TOKEN}"
                },
                json={
                    "replyToken": reply_token,
                    "messages": messages
                }
            )
    return 'OK'

Step 3 上傳程式碼與部署至 Vercel

  1. 將剛才建立的資料夾上傳至你的 GitHub 儲存庫。
  2. 登入 Vercel，點擊 Add New... > Project。
  3. 選擇你剛才上傳的 GitHub 儲存庫，點擊 Import。
  4. 【最重要的一步】 展開 Environment Variables (環境變數) 設定區塊，依序填入 Step 1 準備的 3 把鑰匙：
    Name (名稱)
    Value (數值)
    LINE_CHANNEL_ACCESS_TOKEN
    填入那串超級長的亂碼...
    LINE_CHANNEL_SECRET
    填入32碼英數組合...
    DIFY_API_KEY
    填入 app- 開頭的金鑰...
  5. 點擊 Deploy (部署)。等待畫面灑下紙花，就代表你的伺服器上線了！
  6. 複製 Vercel 派發給你的伺服器網址 (例如：https://my-line-bot.vercel.app)。

Step 4 回到 LINE 設定 Webhook

最後一哩路，我們要讓 LINE 知道有訊息時該往哪裡送：

  1. 填寫 Webhook URL：回到你目前的 LINE 官方帳號管理後台 (Messaging API 設定頁面)。在「Webhook網址」欄位中，貼上你在 Step 3 複製的 Vercel 伺服器網址，並且 一定要在最後面加上 /webhook（因為我們的 Python 程式是接收這個特定路徑）。
    正確格式範例：https://my-line-bot.vercel.app/webhook
  2. 驗證連線與儲存：輸入完畢後點擊「儲存」。你可以點擊下方的 Verify (驗證) 按鈕，如果顯示 Success，恭喜你！橋樑已經接通。
  3. 開啟 Webhook 開關：如果是從 LINE Developers 後台進入，請將 Use webhook 切換為打開狀態。（若在一般管理後台，系統通常會自動啟用）。
  4. 關閉內建自動回應：(非常重要) 根據您的截圖，在「回應設定」頁面下方找到「聊天的回應方式」，將「回應方式」的圓圈改選為 「手動聊天」（也就是取消勾選「手動聊天 + 自動回應訊息」）。這樣就能徹底關閉內建機器人，全權交給 Dify AI 回答！

老師的 Q&A 教室：常見陷阱與疑惑

Q1 關鍵字查詢好難用！難道要把所有問法一條一條建立起來嗎？
A：沒錯，這就是無 AI 狀態下的痛點！ 傳統機器人是「死板的字串比對」，維護成本極高。這也是為什麼我們下一堂課要教大家開啟 Webhook 並導入生成式 AI 的原因。在沒有 AI 之前，建議改用「圖文選單（Rich Menu）」讓使用者用點擊代替打字，避免錯字導致關鍵字失效。
Q2 如果把所有回應都設為「一律回應」，傳一個字會全部一起送出嗎？
A：不會的！LINE 的系統限制同時只能有一則「一律回應」生效。 如果你把 A 和 B 訊息都設為「一律回應」且時間都是「永遠」，系統不會把它們都傳送出去。它通常會根據 「最後修改/存檔時間」 來決定誰是唯一代表。這會導致你的回覆邏輯變得非常混亂。
Q3 那我想要一次回傳很多條內容（如一段文字 + 一張地圖）該怎麼辦？
A：請善用「對話框（泡泡）」功能！ 你不需要建立兩則訊息。而是在「同一則」回應訊息的編輯頁面下方，點擊 「+ 新增」。一則訊息內最多可以串接 5 個對話泡泡。這樣使用者只要觸發一次，就會連續收到文字與地圖。
Q4 「關鍵字回應」和「一律回應」是二選一嗎？
A：不是二選一，而是「優先權」關係。 訊息進來後，系統先檢查有沒有符合的「關鍵字」（第一順位）。如果沒有命中任何關鍵字，才會拋給「一律回應」（第二順位）。因此，正確的架構是：建立多個針對特定資訊的關鍵字回應，並搭配 唯一一個 一律回應來當作安全網。
Q5 程式碼裡面的 user_sessions 是做什麼用的？
A：這是用來讓 AI 「記住你說過的話」的關鍵。 Dify 需要一個叫做 conversation_id 的號碼，才能把對話串接起來。程式碼中建立了一個字典（user_sessions），把每個使用者的 LINE ID 跟他專屬的 conversation_id 綁定在一起。不過要提醒學生，這目前只存在 Vercel 的暫存記憶體裡，如果太久沒用伺服器睡著了，記憶就會重新洗牌喔！
Q6 Dify 出現紅底錯誤「The server encountered an internal error...」怎麼辦？
A：這是因為你的 Dify 機器人「沒有裝上大腦」！

Dify 本身只是一個「管理平台」，它背後需要依靠 OpenAI (ChatGPT)、Anthropic (Claude) 或其他語言模型引擎來運作。畫面右側會提示 「No model provider configured」（未配置模型供應商）。

✅ 解決步驟 (系統模型設定)：

  1. 點擊畫面右側提示的 「Manage models ->」，或是點擊右上角大頭貼進入「設定」>「模型供應商」。
  2. 在列表中找到你要使用的 AI 平台（例如 OpenAI 或 Google），點擊設定並輸入該平台的 API Key（需自行去官網申請）。
  3. 【最重要的一步】 綁定金鑰後，點擊畫面右上角的 「系統模型設定」 按鈕。
  4. 在跳出的對話框中設定：
    • 系統推理模型： 必選！這是機器人對話的大腦（例如選定 Gemini 或 GPT-4o）。
    • Embedding 模型： 若之後要上傳教會講義做「知識庫」，請務必選定此模型（用來將文字轉為向量）。
    • Rerank、語音轉文字等其餘項目，目前可先保持空白。
  5. 點擊對話框右下角的 「儲存」。回到 Agent 的編排畫面，錯誤就會消失了！
Q7 模型供應商畫面顯示的「額度 200」是什麼意思？剩下多少要去哪看？
A：這是 Dify 官方免費送給你的「測試次數」！

為了方便新手直接開始測試，Dify 雲端版會免費贈送 200 次的對話額度。這代表你可以不綁定自己的 API Key，直接免費跟這個 AI 聊 200 句話。

  • 剩下多少去哪看？ 這個畫面上的「200」就是你目前的剩餘次數！當你在 Agent 頁面跟機器人聊天後，回來重新整理這個頁面，數字就會變成 199, 198... 持續遞減。
  • 用完怎麼辦？ 完全不用擔心！只要點擊旁邊的 ⚙️ 配置 按鈕，輸入你自己去 Google AI Studio 或 OpenAI 申請的 API Key，機器人就能繼續運作，而且不再受這 200 次的限制（改為受你自己的 API 帳號額度與計費限制）。
Q8 免費的 Google 帳號申請的 Gemini API Key，額度怎麼算？夠用嗎？
A：非常夠用！這是 Google 給開發者的大放送。

只要你使用一般免費的 Google 帳號（@gmail.com）進入 Google AI Studio 申請 API Key，預設都會套用「免費方案 (Free Tier)」。這對於課堂練習或小型機構來說綽綽有餘。

免費額度限制 (以 Gemini 1.5 Pro/Flash 為例)：

  • RPM (每分鐘請求數)： 15 次 / 分鐘（一分鐘內可以回覆 15 則訊息）。
  • RPD (每天請求數)： 1,500 次 / 天。
  • TPM (每分鐘 Token)： 100萬 Tokens。
教學時務必提醒學生的「兩個雷區」：
  1. 隱私保護： 根據 Google 的政策，使用「免費版 API」傳送的對話內容與檔案，可能會被 Google 收集並由人工審閱來訓練未來的 AI 模型。請強烈告誡學生：絕對不要傳送真實的信徒個資、病歷、或是商業機密！
  2. 短暫罷工： 如果全班同學在同一分鐘內一直狂傳訊息測試（超過 15 次），機器人會因為達到 RPM 上限而短暫「已讀不回」或是報錯。別緊張，等下一分鐘額度重置後就會自動恢復正常了。
